### 1. Defining Backup, Restore

### 1. Defining Backup, Restore, and Recovery

These three terms define the methodology for recovering a database from failure.

| Term | Definition | Key Action | Source Reference |
| :--- | :--- | :--- | :--- |
| **Backup** | The act of creating a means for recovering a database from disaster. It involves making copies of critical physical database files (data files, control files, archived redo log files). | Copy files | |
| **Restore** | The process of reconstructing an Oracle database by copying backup files (data files, control files, etc.) from a secondary storage medium (disk or tape) to their original or new locations on the database server. | Copy back files | |
| **Recover** | The process of updating restored data files by applying redo records from redo logs (online and archived) to bring the database back to a transaction-consistent state, typically to the point where the failure occurred. | Apply redo/changes | |

### 2. Different Types of Backups and Recovery Processes

Database backups are categorized into physical (file copies) and logical (data exports). The core methods for physical backups are Cold and Hot backups.

#### A. Cold Backup (Offline Backup)

A cold backup, or offline backup, is a whole database backup performed while the database is cleanly shut down (`NORMAL`, `IMMEDIATE`, or `TRANSACTIONAL`).

| Feature | Description | Source Reference |
| :--- | :--- | :--- |
| **Consistency** | Produces a **consistent backup** because all files share the same System Change Number (SCN) at the time of shutdown. | |
| **Mode** | Can be used with either **ARCHIVELOG** or **NOARCHIVELOG** mode. | |
| **Recovery Need**| When restored, a cold backup generally does **not require recovery** to open, but is only restored to the point-of-the-last-backup (not the point-of-last-committed-transaction). | |

**Cold Backup Procedure Example (User-Managed, NOARCHIVELOG mode):**

1.  **Shut Down Database:** Shut down the database cleanly using `SHUTDOWN IMMEDIATE`.
    ```sql
    SQL> SHUTDOWN IMMEDIATE;
    ```
2.  **Copy Files:** Copy all required files (data files, control files, online redo log files) using an Operating System (OS) command.
    ```bash
    $ cp /u01/dbfile/o18c/*.* /u01/cbackup/o18c
    ```
3.  **Restore & Open:** The database is restored by copying these files back and can be opened immediately (if online redo logs were included in the backup).

#### B. Hot Backup (Online Backup)

A hot backup, or online backup, is a physical backup performed while the database is open and in use.

| Feature | Description | Source Reference |
| :--- | :--- | :--- |
| **Consistency** | Produces an **inconsistent backup** because the data files may not match the control file SCN since transactions are active. This requires redo application for consistency. | |
| **Mode Requirement**| The database **MUST BE** running in `ARCHIVELOG` mode. | |
| **Recovery Need**| Requires **media recovery** (roll forward using archived redo logs) to bring the restored files to a consistent state. | |

**Hot Backup Procedure Example (User-Managed, Tablespace Level):**

1.  **Set Tablespace Mode:** Place the target tablespace into backup mode. This stops Oracle from recording checkpoint records in the data file headers.
    ```sql
    SQL> ALTER TABLESPACE index01 BEGIN BACKUP;
    ```
2.  **Copy Data Files:** Copy the data files belonging to the tablespace using OS commands.
3.  **End Tablespace Mode:** Take the tablespace out of backup mode. Oracle advances the file headers to the current checkpoint.
    ```sql
    SQL> ALTER TABLESPACE index01 END BACKUP;
    ```
4.  **Control File Backup:** Back up the control file using the `ALTER DATABASE` command.
    ```sql
    SQL> ALTER DATABASE BACKUP CONTROLFILE TO '<filename>';
    ```

**Recovery Process Example (Media Recovery after Hot Backup Loss):**

If a data file from the `index01` tablespace is lost, the recovery (with the database open) proceeds as follows:

1.  **Take Tablespace Offline:** Take the damaged tablespace offline temporarily.
    ```sql
    SQL> ALTER TABLESPACE index01 OFFLINE TEMPORARY;
    ```
2.  **Restore:** Restore the most recent backup of the lost data file using an OS copy command.
3.  **Recover:** Apply archived logs using the `RECOVER` command.
    ```sql
    SQL> RECOVER TABLESPACE index01;
    ```
4.  **Bring Online:** Bring the tablespace back online once recovery is complete.
    ```sql
    SQL> ALTER TABLESPACE index01 ONLINE;
    ```

## 1. Importance of Privileges in Database Security

A **privilege** is a fundamental mechanism of database security. Privileges control user access to the data and restrict the actions users can perform.

The primary importance of privileges in database security is based on the principle of **least privilege**. Oracle recommends granting each user only the privileges required to perform their job, and no more, to ensure greater security and prevent unauthorized activity.

### Types of Privileges:
1.  **System Privileges:** Grant the right to perform a specific action, or to perform an action on *any* schema object of a particular type. For example, the `CREATE TABLE` system privilege allows a user to create tables in their own schema.
2.  **Object Privileges:** Grant the ability to perform a specific action on a **named schema object**. Examples include the right to `SELECT` rows from the `EMPLOYEES` table or `DELETE` rows from the `DEPARTMENTS` table.

## 2. Explanation of Roles in Oracle Database

A **role** is a named group (or collection) of related privileges. Roles are created by users, typically administrators. Unlike schema objects, roles are generally not contained in any user's schema.

### Role in Privilege Management:
Roles are the recommended tool for security administration because they simplify privilege management.

1.  **Reduced Administration:** Instead of granting multiple individual privileges (system and object) to many users, privileges are granted once to a role. The role is then granted to every user needing that access. For instance, if 10 users need access to accounting tables, you grant the privileges to a role and then grant the role to the 10 users.
2.  **Dynamic Updates:** If the required privileges for a group change, only the privileges associated with the role need to be modified. The security domains of all users assigned that role automatically reflect the change.
3.  **Selective Availability:** Roles can be selectively enabled or disabled for a user using the `SET ROLE` statement, allowing specific control over privileges in a given situation.

## 1. What are Data Files?

Data files are the physical components of the database that exist at the operating system level.

*   Data files are the operating system physical files that store the database data on disk.
*   The data is written to these files in an Oracle proprietary format that cannot be read by other programs.
*   Data files have a physical presence and can be viewed outside the database through OS utilities. They persist whether the database is open or closed.
*   **Tempfiles** are a special class of data files associated only with temporary tablespaces.

## 2. How is a Data File Related to a Tablespace?

The relationship is one of **logical container** (tablespace) to **physical storage** (data file).

*   An Oracle database is divided into logical storage units called **tablespaces**, and each tablespace is comprised of one or more operating system files called **data files**.
*   Tablespaces provide the means to physically locate data on storage, as database objects assigned to a tablespace are stored in the physical data files of that tablespace.
*   A data file can be associated with **only one** tablespace and only one database.
*   A segment (representing a database object like a table) can span one or more data files, but it cannot span multiple tablespaces.

## 3. Write a Command to Add a Data File

To increase the storage capacity of an existing tablespace, you use the `ALTER TABLESPACE ... ADD DATAFILE` statement.

This command adds a new data file to the existing tablespace named `USERS`:

```sql
-- Connect as a user with DBA privileges
SQL> ALTER TABLESPACE users
     ADD DATAFILE '/u02/dbfile/o18c/users02.dbf' 
     SIZE 100M;
```

**Explanation:**

*   `ALTER TABLESPACE users`: Specifies the tablespace to be modified.
*   `ADD DATAFILE`: Instructs the database to allocate a new physical file on the operating system.
*   `'/u02/dbfile/o18c/users02.dbf'`: Specifies the full path and filename of the new physical file.
*   `SIZE 100M`: Specifies the initial size of the newly created data file.

## 2. What is Database Auditing? Explain the Standard Auditing in Oracle Database. [1+4]

### What is Database Auditing? (1 Mark)
Database auditing is the process of tracking and recording database activity to ensure security, detect security breaches, and comply with regulatory requirements. It is considered the most accurate record of any database activity. Auditing tracks activities such as the use of privileges, actions of highly privileged users, access to sensitive data, and modifications made to database settings.

### Explanation of Standard Auditing in Oracle Database (4 Marks)

Historically, **Standard Auditing** provided the ability to track activities based on users, privileges, schema objects, and SQL statements. Although traditional auditing is desupported in newer Oracle releases (starting with Oracle AI Database 26ai), modern security relies on the enhanced and unified auditing framework which captures similar activities.

Standard auditing functionality encompasses three main areas:

#### A. Auditing by SQL Statement (Statement Auditing)
This tracks the issuance of a specific SQL statement, such as all `CREATE TABLE` or `ALTER USER` statements. This type of auditing applies only to subsequent sessions, not to current sessions.

*   **Example Commands:**
    *   To audit all successful creations of a table: `AUDIT CREATE TABLE WHENEVER SUCCESSFUL;`.

#### B. Auditing by Schema Object (Object Auditing)
This tracks operations on a specific named object, such as a particular table or index. Object auditing can include DDL or DML statements used on the object. Auditing operations on schema objects apply to current sessions as well as subsequent sessions.

*   **Example Commands:**
    *   To audit all `SELECT` and `INSERT` operations on the `employees` table: `AUDIT SELECT, INSERT ON employees BY <user>;`.

#### C. Auditing by Privilege (Privilege Auditing)
This tracks the issuance of any SQL statement authorized by a specific system privilege granted to a user. Privilege auditing is triggered only if the privileges held by the user are insufficient for the action, meaning the system privilege had to be used to perform the action (e.g., auditing `SELECT ANY TABLE`).

*   **Example Commands:**
    *   To audit when a user attempts to grant or revoke an object privilege: `AUDIT GRANT, REVOKE ON <object> BY <user>;`.

***
The current recommended framework, **Unified Auditing**, uses these standard auditing concepts but consolidates all records (including fine-grained, administrative, and RMAN) into a single, secure repository called the `UNIFIED_AUDIT_TRAIL`. Unified Auditing uses named **Audit Policies** (like `ORA_SECURECONFIG` or custom policies) to group these standard audit settings.

## 3. Explain how SQL Tuning Advisor (STA) can be used to tune your database.

The SQL Tuning Advisor (STA) is a key diagnostic tool used for optimizing the performance and efficiency of SQL statements within the database.

### A. Purpose and Operation

1.  **Objective:** The STA analyzes one or more SQL statements to resolve problems related to sub-optimal performance by suggesting recommendations that optimize the execution plan.
2.  **Execution:** STA runs automatically during the system maintenance windows as part of the Automatic SQL Tuning task. It can also be run manually on demand, typically in response to an ADDM finding.
3.  **Input Sources:** The STA takes high-load SQL statements as input from various sources:
    *   **ADDM (Automatic Database Diagnostic Monitor):** ADDM analyzes AWR data hourly and recommends running STA on high-load SQL statements it identifies as bottlenecks.
    *   **AWR (Automatic Workload Repository):** High-load SQL statements captured in AWR snapshots.
    *   **SQL Tuning Sets (STS):** A set of SQL statements explicitly provided by the user.

### B. Tuning Recommendations

The STA output is in the form of advice, recommendations, rationale, and expected benefits, generated by the Automatic Tuning Optimizer.

The recommendations are used to improve the efficiency of the statement and include the following types of advice:

1.  **Statistics Collection:** Recommending the collection or refreshing of statistics on database objects referenced by the SQL statement.
2.  **Indexing:** Suggesting the creation of new indexes to accelerate data access and query processing.
3.  **SQL Structure:** Advising on restructuring the SQL statement itself for better efficiency.
4.  **SQL Profiles:** Recommending the creation of a **SQL profile**, which is a collection of information that enables the query optimizer to choose a significantly better execution plan at runtime. If Automatic SQL Tuning is enabled for implementation, it can automatically implement SQL profile recommendations (if the benefit is projected to be at least threefold), while other recommendations (like index creation) must be implemented manually.
